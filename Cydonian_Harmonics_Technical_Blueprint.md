## 1. Narrative Architecture and Mathematical Modeling of Harmonic Grid Interactions

The thematic core of *Cydonian Harmonics* centers on a global conflict where ancient human lineages—possessing residual Nephilim genetics—unearth dormant Martian (Cydonian) acoustic monuments on Earth. These monuments generate localized acoustic and electromagnetic waves that can manipulate physical matter, project localized deflector fields, and power automated defense networks.

Instead of classical RTS resource models based on mechanical extraction, gameplay relies on establishing physical resonance networks. Resonance is generated by stationary **Monolith** structures and measured dynamically across the map.

### 1.1 Resonance Field Intensity Model

The spatial intensity \( I \) of a resonance field at any coordinate vector \( \mathbf{x} = (x, y) \) in relation to a source nexus node at \( \mathbf{x}_c = (x_c, y_c) \) is modeled via a damped wave equation that accounts for terrain dissipation and wave phase offset:

\[
I(\mathbf{x}) = I_0 \cdot e^{-\alpha \cdot \|\mathbf{x} - \mathbf{x}_c\|} \cdot \cos\left(\omega \cdot \|\mathbf{x} - \mathbf{x}_c\| + \phi\right)
\]

**Parameters:**

| Symbol | Description |
|--------|-------------|
| \( I_0 \) | Absolute base source intensity |
| \( \alpha \) | Environmental damping coefficient (terrain-dependent) |
| \( \omega \) | Spatial frequency of the Cydonian harmonic propagation |
| \( \phi \) | Node phase alignment offset |
| \( \|\mathbf{x} - \mathbf{x}_c\| \) | Euclidean distance |

The Euclidean distance is computed as:

\[
\|\mathbf{x} - \mathbf{x}_c\| = \sqrt{(x - x_c)^2 + (y - y_c)^2}
\]

### 1.2 Structural Decay Outside Resonance Grids

Units and structures belonging to the Nephilim factions require regular exposure to these fields to operate at peak capacity. When a combat unit travels beyond the boundary of the active resonance grid, its internal structural integrity and shielding decay over time. The structural decay function \( S(t) \) over time \( t \) spent outside an active grid zone is calculated as:

\[
S(t) = S_{\max} \cdot e^{-\lambda \cdot t}
\]

| Symbol | Description |
|--------|-------------|
| \( S_{\max} \) | Maximum defensive shield value |
| \( \lambda \) | Continuous decay constant of the unit's local accumulator |

If a player constructs a defensive relay, the overlapping waves generate localized constructive interference, increasing the effective field intensity where the wave phases align. This mechanic requires players to plan their base layouts carefully, establishing localized relays to extend defensive grids across the game map.

## 2. Technical Foundations of the OpenRA Engine Platform

The OpenRA engine provides a modern, cross-platform framework written in C# that compiles into native assemblies and runs on Microsoft .NET or Mono frameworks. The engine operates on an Entity-Component-System (ECS) design paradigm that separates unit definitions from runtime behaviors.

### 2.1 The ECS Trait Architecture

Within the OpenRA runtime, every game entity—including infantry, heavy vehicles, structures, and projectile effects—is classified as an **Actor**. The functional capabilities of an Actor are determined by its associated **Traits**.

Traits are modular C# components that inherit from a base class and register to receive game engine events. The technical attributes of these traits are declared inside MiniYaml configuration files. During startup, the engine parses these MiniYaml documents using reflection, matching YAML keys directly to the equivalent properties of compiled C# `TraitInfo` classes. This reflection-driven structure eliminates manual registration code and provides direct extensibility.

### 2.2 Sprite Sheets and Render Sheet Constraints

The visual pipeline of *Cydonian Harmonics* uses retro 2D isometric art rendering techniques, drawing frame sequences from compressed sprite sheets and custom paletted formats. The engine relies on custom metadata layers embedded inside standard PNG files to define coordinate maps, frame sequences, and rotation indices.

The primary rendering constraint is the physical size of the texture sheets, which are capped at a maximum of **2048 × 2048 pixels** to ensure compatibility with older GPU drivers.

The packing efficiency \( \eta \) of visual sequences is modeled as:

\[
\eta = \frac{\sum_{i=1}^{F} (w_i \cdot h_i)}{W \times H} \leq 0.85
\]

| Symbol | Description |
|--------|-------------|
| \( F \) | Total count of directional frames |
| \( w_i, h_i \) | Dimensions of each frame |
| \( W \times H \) | Absolute size of the texture sheet |

Exceeding the 0.85 safety threshold can cause texture mapping errors or system out-of-memory issues during large matches. To prevent crashes, multi-directional sequences are split across discrete files, and color maps are mapped to unified 256-color palettes. This allows player colors to be swapped dynamically at index 0 of the paletted image.

## 6. Custom Skills and Isolated Subagents

Custom skills and specialized subagents extend the capabilities of the primary development workspace. Custom skills act as specialized playbooks triggered manually via slash commands. Unlike rules files, which load automatically into the active context, skills are stored in individual folders containing a `SKILL.md` file and any supporting scripts.

### 6.1 Custom Scaffold Skill: `scaffold-trait`

This custom skill automates the boilerplate creation of standard C# traits, using predefined arguments to register appropriate engine event interfaces.

**Skill Metadata:**
- **Name:** `scaffold-trait`
- **Description:** Automatically builds a standardized C# Trait class in the custom assemblies directory, mapping the required OpenRA interface layers.
- **User-invocable:** true

**Scaffolding Protocol:**

1. Verify input parameters before writing files:
   - Trait Name: `$1` (e.g., `CydonianMatterDrift`)
   - Core Event Interface: `$2` (e.g., `ITick`)

2. Write a clean C# code file to `/src/OpenRA.Mods.Cydonian/Traits/`:

```csharp
using System;
using OpenRA.Traits;

namespace OpenRA.Mods.Cydonian.Traits
{
    [Desc("Implements the $1 functionality for Nephilim Chronicles actors.")]
    public class $1Info : TraitInfo
    {
        [Desc("Determines performance multipliers or grid boundaries.")]
        public readonly int AmplitudeMultiplier = 50;

        public override object Create(ActorInitializer init)
        {
            return new $1(init, this);
        }
    }

    public class $1 : $2
    {
        private readonly $1Info info;
        private readonly Actor self;

        public $1(ActorInitializer init, $1Info info)
        {
            this.self = init.Self;
            this.info = info;
        }
    }
}
```

### 6.2 Custom Utility Skill: `lint-rules`

This custom skill provides a standardized workflow to invoke OpenRA's integrated diagnostic linter, catching structural errors across maps, weapons, and sequences.

2. Parse the command output and highlight any lines containing `[YML]` or `[ERR]` errors
3. For simple indentation errors, automatically adjust formatting to conform to the 2-space standard

## 8. Loop Engineering and Verification-Driven Development

Loop engineering shifts the focus from one-shot code generation to automated, iterative verification cycles. Rather than relying on the AI to manually run compiler commands, the workspace uses deterministic PreToolUse and PostToolUse hooks to compile and validate changes automatically. This prevents broken code or invalid rules configurations from being committed to the repository.

### 8.1 The OODA Cycle

The loop follows a highly structured OODA cycle to verify changes before completing a task:

1. **Observe** — The agent scans the workspace to identify compilation errors, syntax issues, or failing tests.
2. **Orient** — The agent analyzes the failures and checks them against technical guidelines or documentation references.
3. **Decide** — The agent plans targeted code corrections or configuration adjustments.
4. **Act** — The agent edits the code and triggers the validation hooks to verify the fix.

This process repeats automatically, ensuring that errors are identified and corrected during the active session without manual intervention.

### 8.2 PreToolUse Guard: `.claude/hooks/guard-pre-tool.py`

This Python script registers as a PreToolUse hook. It intercepts proposed tool calls, verifies commands against standard safety boundaries, and blocks edits to restricted files.

```python
#!/usr/bin/env python3
import sys
import json
import re

def process_pre_tool():
    try:
        # Load the event JSON payload from standard input
        event_payload = json.load(sys.stdin)
    except Exception:
        sys.exit(0)

    tool_name = event_payload.get("tool_name", "")
    tool_input = event_payload.get("tool_input", {})

    # Guard 1: Block potentially destructive system-level bash commands
    if tool_name == "Bash":
        command_string = tool_input.get("command", "")
        unauthorized_regex = [
            r"rm\s+-rf\s+(?!(\.claude/|\.git/|obj/|bin/))",  # Restricted directories
            r"git\s+push\s+.*--force"                        # Prevent forced overwrites
        ]
        for pattern in unauthorized_regex:
            if re.search(pattern, command_string):
                print(json.dumps({
                    "hookSpecificOutput": {
                        "hookEventName": "PreToolUse",
                        "permissionDecision": "deny",
                        "permissionDecisionReason": f"Security Guard Alert: Prohibited pattern detected in command: {pattern}"
                    }
                }))
                sys.exit(0)

    # Guard 2: Enforce read-only locks on environment configurations
    if tool_name in ("Write", "Edit", "MultiEdit"):
        target_file = tool_input.get("file_path", "")
        if re.search(r"(\.env|secrets\.json)$", target_file):
            print(json.dumps({
                "hookSpecificOutput": {
                    "hookEventName": "PreToolUse",
                    "permissionDecision": "deny",
                    "permissionDecisionReason": "Administrative Alert: Directly editing localized environment variables is restricted."
                }
            }))
            sys.exit(0)

    # Let safe actions proceed normally through standard permissions
    sys.exit(0)

if __name__ == "__main__":
    process_pre_tool()
```

### 8.3 PostToolUse Validator: `.claude/hooks/verify-post-tool.sh`

This validation script runs automatically whenever Claude Code writes or edits a file. It parses the stdin JSON payload to find the modified file, maps the file type to the correct compiler or linter, and blocks completion if verification fails.

```bash
#!/usr/bin/env bash
set -euo pipefail

# Parse JSON data from standard input
INPUT_DATA=$(cat)

TOOL_NAME=$(echo "$INPUT_DATA" | jq -r '.tool_name // empty')
FILE_PATH=$(echo "$INPUT_DATA" | jq -r '.tool_input.file_path // empty')

# Immediately exit on standard read-only tool calls
if [[ "$TOOL_NAME" != "Write" && "$TOOL_NAME" != "Edit" ]]; then
    exit 0
fi

if [[ -z "$FILE_PATH" || ! -f "$FILE_PATH" ]]; then
    exit 0
fi

# Run language-specific validation pipelines
case "$FILE_PATH" in
    *.cs)
        echo "Validating C# changes..." >&2
        # Run compiler build command
        if ! make > /dev/null 2> /tmp/compile_errors.log; then
            cat /tmp/compile_errors.log >&2
            echo "Verification Blocked: C# compilation failed." >&2
            # Exit code 2 forces Claude Code to correct the error before proceeding
            exit 2
        fi
        ;;
    *.yaml)
        echo "Validating MiniYaml rules..." >&2
        # Run OpenRA's integrated diagnostic linter
        if ! ./utility.sh cydonian --check-yaml > /tmp/yaml_errors.log 2>&1; then
            cat /tmp/yaml_errors.log >&2
            echo "Verification Blocked: MiniYaml parsing errors identified." >&2
            exit 2
        fi
        ;;
esac

exit 0
```

## 10. Strategic System Operations Protocols

Implementing the architectural layout of *Cydonian Harmonics* requires an orderly, phased deployment to maximize code quality and team velocity:

### Phase 1: Initialize the Standalone SDK
Deploy the OpenRA Mod SDK into the target workspace directory. Run the initial configuration utility `./utility.sh` to fetch and link the core engine dependencies. Immediately create the `CLAUDE.md` and `.claude/settings.json` configurations in the project root to map compiler targets, define file access boundaries, and set default permissions.

### Phase 2: Deploy Automated Validation Hooks
Set up the verification scripts `.claude/hooks/guard-pre-tool.py` and `.claude/hooks/verify-post-tool.sh`. Register these scripts within the "PreToolUse" and "PostToolUse" hooks inside `.claude/settings.json` to ensure every code change undergoes automated compilation, MiniYaml linting, and safety checks before completion.

### Phase 3: Scaffold Custom Mechanics
Use custom skills—such as `scaffold-trait`—to construct C# templates for the dynamic harmonic resonance system. Populate the passive `.claude/docs/` reference directories with detailed formulas, visual standards, and gameplay mechanics, utilizing progressive disclosure with `@`-mentions to keep the active context window clean.

### Phase 4: Scale Production with Parallel Agent Teams
Enable Agent Teams by setting `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`. Define specialized agent profiles with clear file ownership boundaries, allowing narrative writers, sprite animators, and C# engineers to build features concurrently in isolated context windows. Use the shared JSON task list to coordinate tasks and validate components, establishing a highly efficient, verification-driven development pipeline.
