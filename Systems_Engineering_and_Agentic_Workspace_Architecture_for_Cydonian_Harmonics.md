### Mathematical Core — Harmonic Resonance Level

To model the spatial influence of Martian-celestial mechanics on the battlefield, the game employs a mathematical coordinate mapping system representing the influence of the Great Cosmic Clock on tactical gameplay. The level of local spatial corruption and structural reinforcement, known as the **Harmonic Resonance Level** (\( HR \)), is computed continuously across the map grid. This calculation determines the combat efficiency of units and the structural integrity of buildings.

\[
HR(x, y, t) = \sum_{i=1}^{n} \frac{\Phi_i \cdot \Psi_i(t)}{d_i^{2}}
\]

Where:

- \( \Phi_i \) — base tuning frequency of primary transceiver structure \( i \)
- \( \Psi_i(t) \) — dynamic orbital alignment factor of Mars relative to the stellar horizon at time \( t \)
- \( d_i \) — Euclidean distance from the transceiver node to the target coordinate \( (x, y) \)

This dynamic energy field directly scales terrain transit costs, representing the physical distortion caused by the atmospheric interweaving of Mars and Earth. The pathfinding weight calculation is implemented as:

\[
C_{\text{path}}(x, y, t) = C_{\text{base}}(x, y) \cdot \left(1 + \lambda \cdot HR(x, y, t) \cdot e^{-\frac{(x - x_0)^{2} + (y - y_0)^{2}}{2\sigma^{2}}}\right)
\]

Where:

- \( C_{\text{base}}(x, y) \) — baseline pathfinding cost of a terrain cell
- \( \lambda \) — localized resonance decay scalar
- The Gaussian exponential models the spatial falloff of Leyline corruption around a fissure point situated at \( (x_0, y_0) \) with an active alignment radius of \( \sigma \)

## 2. OpenRA Engine Adaptations and Custom C# Core

The technical execution of *Cydonian Harmonics* utilizes the open-source **OpenRA** real-time strategy game engine. OpenRA is written in C# and executes on the .NET framework, providing cross-platform binary compatibility across Windows, macOS, and Linux. The engine decouples the core executable (`OpenRA.Game.exe`) from individual mod files, allowing mods to act as self-contained games. This architecture enables the game to manage its own dependencies, assets, and custom library files, ensuring that engine upgrades do not break the custom gameplay systems.

### Entity-Component-System (ECS) Foundation

The OpenRA engine uses an Entity-Component-System architecture. Every game object is an **Actor** (the entity) composed of a collection of modular behaviors known as **Traits** (the components). The operational parameters of these Traits are defined in text files called **MiniYaml**, which are loaded at runtime by the engine's `FieldLoader` class to populate the corresponding `TraitInfo` definitions.

To implement custom mechanics such as Suntelia Resonance or Nephilim physical scaling, the codebase extends the engine by defining custom C# classes that inherit from `TraitInfo` and implement the corresponding trait logic. These custom components are compiled into a dedicated mod library, `OpenRA.Mods.Cydonia.dll`, which is registered in the main mod manifest file (`mod.yaml`) under the Assemblies list.

### Custom Trait Implementation

```csharp
using System.Collections.Generic;
using OpenRA.Traits;
using OpenRA.Mods.Common.Traits;

namespace OpenRA.Mods.Cydonia.Traits
{
    public class CelestialAlignmentProviderInfo : TraitInfo
    {
        [Desc("Base frequency value generated by this structure.")]
        public readonly int BaseFrequency = 432;

        [Desc("The radius of the Leyline resonance field in cell coordinates.")]
        public readonly WDist ResonanceRadius = new WDist(5120);

        [Desc("Specifies whether this provider requires an active alignment node connection.")]
        public readonly bool RequiresAlignmentNode = true;

        public override object Create(ActorInitializer init)
        {
            return new CelestialAlignmentProvider(init, this);
        }
    }

    public class CelestialAlignmentProvider : ITick, INotifyOwnerCreated
    {
        readonly CelestialAlignmentProviderInfo info;
        readonly Actor self;
        CelestialAlignmentNetwork resonanceNetwork;

        public CelestialAlignmentProvider(ActorInitializer init, CelestialAlignmentProviderInfo info)
        {
            this.self = init.Self;
            this.info = info;
        }

        public void OwnerCreated(Actor self)
        {
            resonanceNetwork = self.World.WorldActor.TraitOrDefault<CelestialAlignmentNetwork>();
        }

        public void Tick(Actor self)
        {
            if (self.IsDead)
                return;

            if (info.RequiresAlignmentNode && !IsConnectedToLeyline())
                return;

            UpdateResonanceFields();
        }

        bool IsConnectedToLeyline()
        {
            return resonanceNetwork != null && resonanceNetwork.IsLeylineActive(self.Position);
        }

        void UpdateResonanceFields()
        {
            var actorsInRadius = self.World.FindActorsInCircle(self.Position, info.ResonanceRadius);
            foreach (var actor in actorsInRadius)
            {
                var receiver = actor.TraitOrDefault<CelestialAlignmentReceiver>();
                if (receiver != null)
                {
                    var distance = (actor.Position - self.Position).Length;
                    var dynamicResonance = info.BaseFrequency / Math.Max(1, (distance * distance));
                    receiver.ApplyHarmonicBoost(dynamicResonance);
                }
            }
        }
    }

    public class AnakimPhysicalPresenceScalerInfo : TraitInfo
    {
        [Desc("Base multiplier for the dynamic size scaling of the giant Nephilim unit.")]
        public readonly int VisualScaleFactor = 300;

        public readonly int CustomCollisionRadius = 768;

        public override object Create(ActorInitializer init)
        {
            return new AnakimPhysicalPresenceScaler(init, this);
        }
    }

    {

        readonly Actor self;

        public AnakimPhysicalPresenceScaler(ActorInitializer init, AnakimPhysicalPresenceScalerInfo info)
        {
            this.self = init.Self;
            this.info = info;
        }

        public void Created(Actor self)
        {
            ApplyDynamicPresenceScaling();
        }

        void ApplyDynamicPresenceScaling()
        {

    {

        {
```

### Modular Rules (`.cursor/rules/`)

Rules are Markdown files with YAML frontmatter containing three key fields:

- `alwaysApply` (bool)
- `globs` (array of path patterns)
- `description` (semantic string used to route intelligent model requests)

The system uses this metadata to load rules dynamically, preventing context overload.

#### Example: OpenRA Trait Development Standards

```markdown
---
description: Enforces high-performance, non-blocking ECS architecture patterns for C# trait development.
globs: ["OpenRA.Mods.Cydonia/**/*.cs"]
alwaysApply: false
---

# OpenRA Trait Development Standards

## Performance Constraints
1. Avoid inline allocation during tick operations (`ITick.Tick`). Use cached arrays or structures.
2. Implement `INotifyOwnerCreated` to fetch external dependency traits on startup instead of using `self.TraitOrDefault<T>()` inside loops.
3. Use safe code blocks unless interacting directly with native graphics pointers.

## Architecture Bindings
1. Every trait must provide a clean matching `Info` class descending from `TraitInfo`.
2. Map settings dynamically using the OpenRA `[FieldLoader]` and `[Desc]` attributes for integration with MiniYaml files.
```


### Skill Specifications

#### Sprite Asset Processing and Palette Sheet Mapping Skill

**Context**  
Use this skill when processing raw PNG sheets into OpenRA-compatible palette textures, sequences, and SHP frame assets.

**Execution Workflow**
1. Read the input spritesheet PNG. Validate that the transparent background color is matched to Index 0.
2. Translate raw textures to the standard Red Alert or Tiberian Dawn palette using `OpenRA.Utility.exe cydonia --remap-palette`.
3. Scaffold the corresponding definitions in `mods/cydonia/sequences.yaml` matching the output frame counts.

**Recovery Operations**
1. If the utility returns a "palette out of bounds" error, parse the input palette indexes and locate coordinates exceeding Index 255.
2. If frames render with corrupted noise, check that image borders align exactly with the cell size dimensions defined in the sequences configuration.

#### Lore Verification and Nomenclature Alignment Skill

**Context**  
Use this skill when generating text scripts, mission dialogues, unit descriptions, and audio trigger scripts to ensure they match canonical Enochian lore.

**Execution Workflow**
1. Intercept any written text generated for user-facing units or structures.
2. Query `.cursor/docs/lore-lexicon.md` using semantic text filters.
3. Compare proposed names against the approved list of entities (e.g., Semjâzâ, Azâzêl, Gregori, Anakim, Mount Sinai).
4. Reject modern or generic terminology and replace with canonical equivalents.

**Recovery Operations**
1. If a term is flagged as "non-canonical," retrieve the nearest thematic equivalent from the lexicon and output an error log indicating the correction.

## 5. Autonomic Loop Engineering and Verification

The workspace is configured with an automated feedback loop that manages compiling, asset validation, and regression testing. This cycle uses the `.cursor/hooks.json` file to run verification scripts during development, automating the process of resolving syntax and logical errors.

```json
{
  "version": 1,
  "hooks": {
    "afterFileEdit": [
      {
        "command": "bun run .cursor/hooks/grind.ts"
      }
    ]
  }
}
```

The `grind.ts` script runs under the Bun runtime after every file modification. It compiles the custom C# libraries, runs the OpenRA asset checks, and feeds any compile errors back into the agent context, repeating the cycle until the build is clean.

```typescript
import { spawn } from "child_process";
import * as fs from "fs";

interface CompileResult {
  success: boolean;
  errorLog?: string;
}

async function runBuild(): Promise<CompileResult> {
  return new Promise((resolve) => {
    const makeProcess = spawn("make", ["all"], { shell: true });
    let stderrData = "";
    let stdoutData = "";

    makeProcess.stdout.on("data", (data) => {
      stdoutData += data.toString();
    });
    makeProcess.stderr.on("data", (data) => {
      stderrData += data.toString();
    });

    makeProcess.on("close", (code) => {
      if (code === 0) {
        resolve({ success: true });
      } else {
        resolve({ success: false, errorLog: stderrData || stdoutData });
      }
    });
  });
}

async function runLinter(): Promise<CompileResult> {
  return new Promise((resolve) => {
    const utilityProcess = spawn("utility.sh", ["cydonia", "--lint"], { shell: true });
    let output = "";

    utilityProcess.stdout.on("data", (data) => { output += data.toString(); });
    utilityProcess.stderr.on("data", (data) => { output += data.toString(); });

    utilityProcess.on("close", (code) => {
      if (code === 0) {
        resolve({ success: true });
      } else {
        resolve({ success: false, errorLog: output });
      }
    });
  });
}

async function main() {
  const buildStatus = await runBuild();
  if (!buildStatus.success) {
    const failurePayload = {
      status: "failed",
      stage: "compilation",
      error: buildStatus.errorLog,
      followup_message: `The project compilation failed with the following errors. Please locate the source files and resolve the syntax or binding mismatches: \n\n${buildStatus.errorLog}`
    };
    console.log(JSON.stringify(failurePayload));
    process.exit(1);
  }

  const lintStatus = await runLinter();
  if (!lintStatus.success) {
    const failurePayload = {
      status: "failed",
      stage: "linting",
      error: lintStatus.errorLog,
      followup_message: `The OpenRA rules linter detected semantic inconsistencies in the MiniYaml structure: \n\n${lintStatus.errorLog}`
    };
    console.log(JSON.stringify(failurePayload));
    process.exit(1);
  }

  console.log(JSON.stringify({ status: "completed", message: "Build and validation checks completed cleanly." }));
  process.exit(0);
}

main();
```

## 7. Mod SDK Build Pipeline and Release Packaging

The final step in the mod development process compiles the custom assets, rules, and logic into standalone installers for players. The decoupled Mod SDK resolves historical version-mismatch issues by packaging the game engine alongside the custom mod files, producing isolated installers that run independently of other OpenRA games.

![Build Pipeline](images/build_pipeline.png)

*Figure 5. Automated packaging pipeline controlled by `mod.config` and orchestrated by `packaging/package-all.sh`.*

### Core Configuration (`mod.config`)

```bash
# Core Mod SDK Configuration Variables
ENGINE_VERSION="playtest-20210131"
AUTOMATIC_ENGINE_MANAGEMENT="True"
AUTOMATIC_ENGINE_SOURCE="https://github.com/OpenRA/OpenRA/archive/${ENGINE_VERSION}.zip"
PACKAGING_COPY_MOD_BINARIES="OpenRA.Mods.Cydonia.dll"
PACKAGING_COPY_ENGINE_FILES="True"
PACKAGING_OSX_MONO_TAG="osx-launcher-20200830"
PACKAGING_APPIMAGE_DEPENDENCIES_TAG="20200328"
```

### Packaging Sequence

1. **Clean Workspace Targets** — Clears previous build artifacts and resets temporary directories.
2. **Fetch Core Engine Artifacts** — Downloads the target engine zip file and extracts it to the assembly cache.
3. **Compile Library Binaries** — Invokes the project's compiler via `make` to compile the custom traits into `OpenRA.Mods.Cydonia.dll`.
4. **Run Asset Verification** — Runs the OpenRA linter to verify the validity of custom maps, sequences, and rule definitions.
5. **Package for Windows** — Runs the NSIS compiler (`makensis`) to generate a Windows installer including the engine files, C# library, and custom assets.
6. **Package for macOS** — Generates an `.app` bundle containing the engine files, compiled traits, and custom assets.
7. **Package for Linux** — Packages the game and its dependencies into a single, self-contained AppImage bundle.

This decoupled structure ensures that *Cydonian Harmonics* remains stable and independent of updates to other OpenRA games on the player's system.

## 8. Technical Conclusions and Actionable Synthesis

Synthesizing the game systems and automated workspace integration highlights the key steps required to successfully build and deploy *Cydonian Harmonics*:

1. **Maintain Trait Safety and Performance**  
   When writing custom C# traits (such as `CelestialAlignmentProvider` or `AnakimPhysicalPresenceScaler`), optimize the code to avoid runtime memory allocations during tick loops. Retrieve internal components during initialization using `INotifyOwnerCreated` rather than relying on on-demand lookups during combat simulations.

2. **Implement Structural Decoupling**  
   Keep game systems and developer rules separate by mapping file patterns directly in `.cursor/rules/openra-csharp-standards.mdc` and `.cursor/rules/yaml-rules-linting.mdc`. This approach prevents context bloat, reduces token costs, and ensures that model suggestions remain accurate across different files.

3. **Automate Compilation and Linting**  
   Set up automatic validation loops using `hooks.json` and a background `grind.ts` script. Running asset checks and compilation verification immediately after code changes catches errors early and helps autonomous developer agents resolve issues without manual intervention.

4. **Use Independent Faction Paths**  
   Map the distinct thematic differences between the Celestial Watchers and the Antediluvian Dynasties to separate gameplay systems, using custom traits instead of standard tech trees. This strategy keeps tactical balance adjustments isolated and easy to manage.

5. **Standardize Build Settings**  
   Maintain the mod's target dependencies and assembly output configurations directly in `mod.config`. Using isolated build processes for each target platform ensures consistent gameplay performance and clean installations on Windows, macOS, and Linux.
